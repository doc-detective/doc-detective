# Content strategy

This directory holds the durable content strategy for the Doc Detective documentation site. It is the
reference every writing task should consult before drafting or restructuring a page.

These files live inside `docs/` but **outside** `docs/fern/pages/**`, so they are not published as
end-user pages. They are internal working documents for agents and contributors.

## Files

| File | Contents |
|---|---|
| `audiences.md` | The five target audiences and why each matters. |
| `personas.md` | Full profiles for Wren, Diego, Priya, Aria, and Cole. |
| `cujs.md` | Critical User Journeys per persona — the end-to-end outcomes the docs must support. |
| `information-architecture.md` | The CUJ-based IA and nav tree, plus the full content-set map (which page serves which CUJ, with ★ launch markers) and the source-of-truth mapping for reference pages. |

## How to use this during writing tasks

Before drafting or editing any user-facing documentation:

1. **Identify the relevant persona.** Is this page for Wren (documentation engineer, lead), Diego
   (developer / API tester), Priya (CI / platform engineer), Aria (AI-assisted author), or Cole
   (contributor)? A page may serve more than one, but there is usually a primary.

2. **Find the matching CUJ in `cujs.md`.** Each persona has a small set of numbered journeys
   (W1–W3, D1–D3, P1–P3, A1–A2, C1), plus the cross-cutting X1. Understand the end-to-end outcome the
   persona needs to reach.

3. **Structure content around that journey, not by document type.** Do not impose a Diátaxis-style
   tutorial/how-to/explanation/reference split as the organizing principle. Ask: "What does this
   persona need to know, and in what order, to reach the outcome?" Let the journey sequence the
   content.

4. **Link into the Reference shelf for lookups.** Full action fields, the complete `config_v3` key
   list, every CLI flag, and the context/selector tables belong in `reference/`. Journey pages explain
   the path and link into reference for exhaustive detail — they do not duplicate it.

5. **Check the IA map.** `information-architecture.md` lists every planned page, the CUJ it serves, and
   whether it is a ★ launch priority. If you are adding a new page, record it there.

6. **Frontmatter.** Every page in `docs/fern/pages/**` must have `title` and `description` in its
   frontmatter, per [`docs/AGENTS.md`](../AGENTS.md). Use a short `sidebar` title when the title is
   long. No level-1 (`#`) Markdown headings; use the frontmatter title.

## Verifying technical claims

Doc Detective docs document a real CLI, schema set, and runner, so every flag, action field, config
key, and exit behavior must match the code — never the writer's assumption. Reference pages name their
**source of truth** in `information-architecture.md`; cross-read it before writing:

- **Flags and commands:** `buildYargs()` in [`src/utils.ts`](../../src/utils.ts) and
  [`src/cli.ts`](../../src/cli.ts).
- **Config keys:** [`config_v3.schema.json`](../../src/common/src/schemas/src_schemas/config_v3.schema.json)
  and [`src/core/config.ts`](../../src/core/config.ts).
- **Actions / step fields:** the per-action schemas under
  [`src/common/src/schemas/src_schemas/`](../../src/common/src/schemas/src_schemas/) and the
  implementations in [`src/core/tests/`](../../src/core/tests/).
- **Run output and reporters:** [`src/reporters/`](../../src/reporters/).

When in doubt, run the tool against a fixture and quote the real output rather than hand-writing it.
The docs test themselves (`npm run doc-detective` in `docs/`), so claims that drift from behavior break
CI.

### Generated reference pages — never hand-edit

The schema reference pages under `docs/fern/pages/reference/schemas/` (e.g. `config.md`, `context.md`,
and every per-action schema page) are **generated by a script** from the `doc-detective-common`
schemas — currently
[`.scripts/buildSchemaReferencesV4.js`](https://github.com/doc-detective/doc-detective.github.io/blob/main/.scripts/buildSchemaReferencesV4.js).
To change one, edit the JSON schema's `description` / `default` / `examples` / `enum` in
[`src/common/src/schemas/src_schemas/`](../../src/common/src/schemas/src_schemas/) and re-run the
generator — **do not edit the `.md` by hand**; it will be overwritten. The hand-written
`docs/fern/pages/docs/actions/*.mdx` prose pages are *not* generated and are edited directly. See the
"Generated reference pages" note in `information-architecture.md` for the full rule and the open
infrastructure gap (the generator isn't yet ported into this repo's Fern path).
