---
title: startSurface
description: Open one or more surfaces—native apps, browsers, or background processes—and register them by name.
---


The `startSurface` action opens (provisions) a surface and registers it by name so later steps can target it with `surface`. It handles three kinds of surface, told apart by which key the descriptor carries:

- **App** (`app`): launch a native application—a Windows or macOS desktop app, or an Android/iOS app on a managed device.
- **Browser** (`browser`): open a browser session explicitly, with per-session launch options. The session opens blank and ready. Navigate it with [`goTo`](goTo).
- **Process** (`process`): start a long-running background process—first-class sugar over [`runShell`](/docs/actions/runshell) with a `background` object. Both forms share one launcher.

Pass a single descriptor to open one surface, or an **array of descriptors to open several concurrently**—browser and process descriptors launch in parallel, and app descriptors boot their devices in parallel (the sessions themselves open in authored order). One step can boot two emulators for a two-phone conversation instead of waiting out two serial boots.

<Note>
Doc Detective drives native desktop apps through a platform-specific Appium driver — `appium-novawindows-driver` on Windows, `appium-mac2-driver` on macOS — and installs it on demand the first time an app surface starts. To install one ahead of time, run `doc-detective install runtime appium-novawindows-driver` on Windows or `doc-detective install runtime appium-mac2-driver` on macOS.
</Note>

## App descriptors

- `app`: (Required) The app identifier. Doc Detective identifies the kind from the value's syntax, never from a separate type field:
  - An **executable path** (Windows), such as `C:\Windows\System32\charmap.exe`.
  - A **UWP AppUserModelID** (Windows), such as `Microsoft.WindowsCalculator_8wekyb3d8bbwe!App`.
  - A **`.app` path** (macOS), such as `/System/Applications/Calculator.app`.
  - A **bundle ID** (macOS), such as `com.apple.TextEdit`.
  - A **package name** (Android), such as `com.example.myapp`.
- `name`: (Optional) The surface-registry name that later steps use in `surface`. If you omit it, Doc Detective derives a default: the executable file name without its extension (`charmap.exe` becomes `charmap`), the base name of a `.app` path (`/System/Applications/Calculator.app` becomes `Calculator`), the final dot-segment of a bundle ID (`com.apple.TextEdit` becomes `TextEdit`), or the app token of an AppUserModelID (`Microsoft.WindowsCalculator_8wekyb3d8bbwe!App` becomes `WindowsCalculator`). Doc Detective reserves the browser engine keywords `chrome`, `firefox`, `safari`, `webkit`, and `edge`. They can't name an app surface, because a bare-string `surface` set to one of them always targets that browser. The same holds for a derived default: launching `chrome.exe` without a `name` derives `chrome`, so the step fails and asks you to pass a non-keyword `name`. The check is case-insensitive, so it rejects `Chrome` too.
- `args`: (Optional) An array of launch arguments to pass to the app. On macOS they reach the app as a real argument array. On Windows they join into a single shell-style argument string for the driver, so an argument that contains spaces must carry its own quotes.
- `env`: (Optional) Extra environment variables for the launched app. Supported on macOS. The Windows driver doesn't support `env`, and setting it there fails the step with guidance. Instead, set variables in the shell that launches Doc Detective, or launch the app with [`runShell`](/docs/actions/runshell).
- `workingDirectory`: (Optional) The working directory for the launched app. Defaults to the run's working directory. No current desktop app driver honors it—on Windows the NovaWindows driver ignores it, and macOS launches apps through LaunchServices, which offers no working-directory control—so any non-default value fails the step with guidance rather than being silently ignored. Use [`runShell`](/docs/actions/runshell) if the working directory matters.
- `install`: (Optional, mobile) An `.apk`/`.app`/`.ipa` to install before launch.
- `device`: (Optional, mobile) The managed device to run on—a name, or a descriptor with `platform`, `name`, `deviceType`, `osVersion`, `headless`.
- `waitUntil`: (Optional) A readiness condition that must hold before Doc Detective considers the surface open. Accepts `delayMs` (wait at least this many milliseconds) and/or `find` (an element that must exist). No condition applies by default. See [Wait for the app to be ready](#wait-for-the-app-to-be-ready).
- `timeout`: (Optional) The startup ceiling in milliseconds, covering launch and readiness. Defaults to `60000`. On macOS, the first app surface of a run also builds the automation agent (WebDriverAgentMac) with Xcode, which can take several minutes on a cold machine; Doc Detective raises the server startup ceiling to at least 120000 ms for that first build regardless of `timeout`.
- `driverOptions`: (Optional) An escape-hatch object merged into the automation session's capabilities after the ones Doc Detective computes. Capabilities are driver- and version-specific and prefixed per driver, for example `{ "appium:noReset": true }`. Use it sparingly.

## Browser descriptors

- `browser`: (Required) Engine: `chrome`, `firefox`, `safari`, `webkit`, or `edge`. Opens a blank, ready session on the context's automation server. No cross-engine fallback—you named the engine, so Doc Detective never substitutes another.
- `name`: (Optional) Surface-registry name. Default: the engine keyword. The context's default browser already registers under its engine name, so a second session of the same engine needs an explicit `name` (an unnamed duplicate fails with guidance).
- `headless`: (Optional) Overrides the context's headless setting for this session.
- `size`: (Optional) Outer window dimensions (`{ "width": …, "height": … }`).
- `viewport`: (Optional) Page-content dimensions, which win over `size`.
- `driverOptions`: (Optional) Escape-hatch capability passthrough.

There's deliberately no `url` on the browser descriptor—[`goTo`](goTo) stays the navigation step—and no `waitUntil`: a browser session is ready the moment it opens.

## Process descriptors

- `process`: (Required) The shell command, same semantics as [`runShell`](/docs/actions/runshell).
- `name`: (Required) Surface-registry name. Required because background commands are generic interpreters (`node`, `docker`) whose derived defaults would collide.
- `args`, `workingDirectory`: (Optional) Command arguments and working directory.
- `tty`: (Optional) Spawn under a pseudo-terminal for full-screen/interactive TUIs (needs the optional `node-pty` dependency, whose absence skips the step).
- `waitUntil`: (Optional) Readiness: `port`, `stdio` (substring or `/regex/`), `httpGet`, and/or `delayMs`. Same conditions as runShell's `background.waitUntil`.
- `timeout`: (Optional) Readiness ceiling in milliseconds (default 60000).

## Parallel arrays

Pass an array to open several surfaces in one step. Names must be unique within the array (a duplicate fails the step before anything launches). The step passes only when every descriptor opened. One failure fails the step, but the other surfaces still finish opening and appear in `outputs.surfaces`, so nothing leaks half-started. After a parallel open, the **last authored** descriptor of each kind is that kind's active surface—deterministic regardless of which lane finished first.

> For comprehensive options, see the [`startSurface`](/reference/schemas/startsurface) reference.

## Target an app surface

After `startSurface` registers an app, reference it from a later step with a `surface` object that names it:

```json
{
  "surface": { "app": "notepad" }
}
```

Doc Detective locates elements on an app surface through the platform's accessibility layer (UI Automation on Windows, the macOS accessibility (AX) tree on macOS), so the element criteria you already use in the browser map onto native controls:

- `elementText` matches a control's visible text. On macOS this covers a control's title, label, or value, so it also matches the content of text views and value displays.
- `elementId` and `elementTestId` match a control's automation ID (its accessibility identifier on macOS).
- `elementAria` matches a control's role, and its accessible name when you provide one. Pass the accessible name as a string; the `{ role, name }` object form isn't available yet.

For controls the semantic fields can't reach, `selector` accepts a native locator instead of a CSS selector:

- An **XPath**. On Windows the element names are UIA control types, such as `//Button[@Name="Save"]`; on macOS they're XCUIElementType tags, such as `//XCUIElementTypeButton[@title="Add"]`.
- An **accessibility ID**, prefixed with `~`, such as `~SaveButton`.

CSS selectors are browser-only and fail on app surfaces, as do the `elementClass` and `elementAttribute` fields. The error message names the supported alternatives.

## Platform support

`startSurface` launches native desktop apps on Windows and macOS. On Linux, the app preflight skips the context with a reason rather than failing, which keeps a mixed-platform spec passing where it can't run.

Because an app targets one platform, gate app tests to the platform they run on with a [context](/docs/test-docs/platforms-and-browsers). A Windows app targets the `windows` platform:

```json
{
  "runOn": [
    {
      "platforms": ["windows"]
    }
  ]
}
```

A macOS app targets the `mac` platform:

```json
{
  "runOn": [
    {
      "platforms": ["mac"]
    }
  ]
}
```

You can also require the app's binary so the context skips cleanly on machines that don't ship it, using the context [`requires`](/docs/test-docs/platforms-and-browsers) capability gate:

```json
{
  "runOn": [
    {
      "platforms": ["windows"],
      "requires": {
        "files": ["C:\\Windows\\System32\\charmap.exe"]
      }
    }
  ]
}
```

### Grant Accessibility permission on macOS

macOS gates app automation behind the Accessibility (TCC) permission. The process that runs Doc Detective (your terminal, IDE, or CI runner process) needs permission to control your computer, or the app driver can't drive apps.

Before launching an app, Doc Detective checks whether you granted the permission. If macOS reports it as denied, the app preflight skips the context with a walkthrough rather than failing:

> Open System Settings → Privacy & Security → Accessibility and enable the app that launches Doc Detective (your terminal, IDE, or CI runner process), then rerun.

Grant the permission there, then rerun. If the launch still fails with an accessibility-related error, Doc Detective repeats the same walkthrough in the failure message.

## Wait for the app to be ready

Launching an app returns before its window is interactive. Use `waitUntil` to hold the step until the app is ready, so the next step doesn't run against a window that hasn't drawn its controls yet:

- `delayMs`: Wait at least this many milliseconds.
- `find`: Wait until an element exists on the app surface, using the same criteria as a `find` step.

`timeout` caps the combined launch-and-readiness wait. If the readiness element never appears before `timeout` elapses, the step fails and reports that the app launched but never became ready.

## Close an app surface

Close an app surface with [`closeSurface`](/docs/actions/closesurface), referencing it by name:

```json
{
  "closeSurface": { "app": "notepad" }
}
```

You don't have to close every app explicitly. Doc Detective closes any app surfaces that are still open when the run finishes.

## Examples

### Open a native app

```json
{
  "tests": [
    {
      "steps": [
        {
          "startSurface": {
            "app": "C:\\Windows\\System32\\charmap.exe",
            "waitUntil": { "find": { "elementText": "Select" } }
          }
        },
        {
          "click": { "elementText": "Select", "surface": { "app": "charmap" } }
        }
      ]
    }
  ]
}
```

### Launch a macOS app and act on it

This example launches Calculator by its `.app` path, waits until a digit button exists, computes `1 + 2`, checks that the display reads `3`, captures the window, and closes the surface. The digit buttons match by accessible name with `elementAria`. The operator keys use a native XPath selector because their accessible names are words (`Add`, `Equals`) rather than the symbols on screen.

```json
{
  "tests": [
    {
      "runOn": [
        {
          "platforms": ["mac"]
        }
      ],
      "steps": [
        {
          "description": "Launch Calculator and wait until it's ready.",
          "startSurface": {
            "app": "/System/Applications/Calculator.app",
            "waitUntil": {
              "find": { "elementAria": "1" }
            }
          }
        },
        {
          "description": "Press 1.",
          "click": {
            "elementAria": "1",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Press the add key.",
          "click": {
            "selector": "//XCUIElementTypeButton[@title=\"Add\" or @label=\"Add\"]",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Press 2.",
          "click": {
            "elementAria": "2",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Press the equals key.",
          "click": {
            "selector": "//XCUIElementTypeButton[@title=\"Equals\" or @label=\"Equals\"]",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Check the result.",
          "find": {
            "elementText": "3",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Capture the window.",
          "screenshot": {
            "path": "calculator-app.png",
            "surface": { "app": "Calculator" }
          }
        },
        {
          "description": "Close Calculator.",
          "closeSurface": { "app": "Calculator" }
        }
      ]
    }
  ]
}
```

### Open a named browser session with launch options

```json
{
  "tests": [
    {
      "steps": [
        {
          "startSurface": {
            "browser": "firefox",
            "name": "admin",
            "headless": true,
            "viewport": { "width": 800, "height": 600 }
          }
        },
        {
          "goTo": {
            "url": "https://example.com/admin",
            "surface": { "browser": "firefox", "name": "admin" }
          }
        }
      ]
    }
  ]
}
```

### Start a background server and call it

```json
{
  "tests": [
    {
      "steps": [
        {
          "startSurface": {
            "process": "npm start",
            "name": "web",
            "waitUntil": { "port": 3000 }
          }
        },
        {
          "httpRequest": {
            "url": "http://localhost:3000/health",
            "statusCodes": [200]
          }
        },
        {
          "closeSurface": "web"
        }
      ]
    }
  ]
}
```

### Open several surfaces concurrently

Two managed Android devices boot in parallel—the "two-phone conversation" pattern:

```json
{
  "tests": [
    {
      "steps": [
        {
          "startSurface": [
            {
              "app": "com.example.chat",
              "name": "phone-a",
              "device": { "platform": "android", "name": "dd-phone-a" }
            },
            {
              "app": "com.example.chat",
              "name": "phone-b",
              "device": { "platform": "android", "name": "dd-phone-b" }
            }
          ]
        },
        {
          "find": { "elementText": "Sign in", "surface": { "app": "phone-a" } }
        }
      ]
    }
  ]
}
```

Kinds mix freely in one array—a browser and the server it talks to:

```json
{
  "startSurface": [
    { "process": "npm start", "name": "web", "waitUntil": { "port": 3000 } },
    { "browser": "chrome", "name": "shopper" }
  ]
}
```
