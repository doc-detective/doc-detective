title: record
description: Start capturing a video of test execution.
---


The `record` action starts recording test execution as a video file (MP4, WebM, or GIF). This is useful for debugging failed tests or creating multimedia documentation. Recordings are video only, with no audio track. Recording continues until your test reaches a `stopRecord` step.

<Note>
Recording works with concurrent test runners (`concurrentRunners` > 1). The `browser` engine records each context's own window, so it always runs concurrently. The `ffmpeg` engine captures the whole shared display, so Doc Detective serializes `ffmpeg` recordings with other browser-based tests automatically—you don't need to set `concurrentRunners` to `1`. See [Recording engines](#recording-engines) for details.
</Note>

You can specify the recording action in several ways:

- **Boolean Shorthand:** Set `record: true` to start recording with default settings (usually saved as `record_<timestamp>.mp4` in the output directory).
- **String Shorthand:** Provide a file path directly as the value for the `record` key (for example, `record: "test_session.webm"`). This starts recording to the specified path. Supported extensions are `.mp4`, `.webm`, and `.gif`.
- **Object Format:** Use an object for more control over the output file:
  - `path`: (Optional) The file path where the recording will be saved. Can include directories. If omitted, a default path is generated.
  - `directory`: (Optional) The directory where the recording will be saved. If `path` includes a directory, this is ignored. Defaults to the configured output directory.
  - `overwrite`: (Optional) Set to the string `"true"` to overwrite an existing file at the target path. Defaults to `"false"`.
  - `name`: (Optional) An identifier for this recording. A later `stopRecord` step can target it by name to stop a specific recording when several overlap. See [Multiple overlapping recordings](#multiple-overlapping-recordings) below.
  - `engine`: (Optional) The recording engine to use. See [Recording engines](#recording-engines) below.
  - `surface`: (Optional) The surface to record. For browsers, a session, window, or tab reference, for example `{ "browser": "chrome", "tab": "demo" }`; omit to record the active tab. The recorder's own internal tab is never addressable by window/tab selectors. See [Test across multiple tabs, windows, and browsers](/docs/test-docs/multiple-tabs-and-windows). For native apps, the object form `{ "app": "<surface name>" }` records that app, cropped to its window by default. See [Recording app surfaces](#recording-app-surfaces).

**Stopping the Recording:** Include a `stopRecord` step to end a recording at a specific point in your test. Doc Detective automatically finalizes any remaining recordings at the end of the test run.

## Recording engines

Recording uses one of two engines, selected by the optional `engine` field in the object format:

- **`browser`:** Records the Chrome viewport using the browser's screen-capture API. It captures only the current browser tab and is safe to run with concurrency (when `concurrentRunners` is greater than 1), because each context records its own window. The `browser` engine requires a visible Chrome context and can't record in headless mode.
- **`ffmpeg`:** Records the screen with FFmpeg, so it works for any application, not just Chrome. Doc Detective installs FFmpeg automatically, so you don't need to install it yourself.

If you don't set `engine`, Doc Detective chooses one for you. It prefers the `browser` engine when a visible Chrome context is available (coercing an unspecified browser to headed Chrome), and falls back to `ffmpeg` otherwise.

Set `engine` as a string to select an engine with its defaults (for example, `"engine": "ffmpeg"`), or as an object for more control:

- `name`: (Required) The engine to use, either `browser` or `ffmpeg`.
- `target`: (Optional, `ffmpeg` only) What the engine captures. Use `display` to record the full screen, `window` to record the active window, or `viewport` to record the browser content area. If unset, the target defaults to `window` when the recording's `surface` is an app surface and to `display` otherwise. The `window` and `viewport` targets are best-effort: FFmpeg captures the full screen and then crops the result. The `browser` engine ignores this field and always captures its tab.
- `fps`: (Optional, `ffmpeg` only) The capture frame rate (default: `30`).

On Android and iOS contexts, `engine` doesn't apply. Doc Detective records the device screen through the device itself. See [Recording on mobile devices](#recording-on-mobile-devices).

**Recording concurrency** depends on the engine. The `browser` engine is always safe to run concurrently, because each context records its own window.

The `ffmpeg` engine captures the whole shared display, so two `ffmpeg` recordings—or an `ffmpeg` recording alongside another browser-driven test—would overwrite each other's output. To prevent this, whenever a run includes an `ffmpeg` recording, Doc Detective serializes that recording with every other browser-based test so they take turns on the display. Jobs that don't drive a browser, such as `httpRequest` and `runShell` steps, keep running in parallel up to `concurrentRunners`. This behavior applies on every platform, including Linux. To run the recordings concurrently as well, use the `browser` engine instead.

On Linux, the `ffmpeg` engine also needs Xvfb to record headless contexts, which have no physical display to capture. Install Xvfb to record in headless mode.

## Recording app surfaces

A `record` step can target a native app surface opened by [`startSurface`](/docs/test-docs/test-apps) with the object form `{ "app": "<surface name>" }`:

```json
{
  "record": {
    "path": "notepad-demo.mp4",
    "surface": { "app": "notepad" }
  }
}
```

App-surface recordings use the `ffmpeg` engine with a `window` target by default. FFmpeg captures the full display, and when the recording stops, Doc Detective crops the video to the app's window—including on scaled (HiDPI/Retina) displays, where Doc Detective scales the crop to match the capture's physical pixels. Set `engine: { "name": "ffmpeg", "target": "display" }` to keep the full display instead. The `viewport` target is the browser content area, so it doesn't apply to app surfaces—Doc Detective skips a record step that combines them and explains why, and does the same for the `browser` engine (it records a Chrome tab, not a native window).

To record a specific window of the app—a dialog, or one of several document windows—add a `window` selector: `{ "app": "notepad", "window": { "title": "/Open/" } }`. The crop then bounds that window. See [App windows](find#app-windows) for the selector grammar.

App-surface recordings join the same display serialization as other FFmpeg recordings, and the same overlapping/named-stop rules apply.

Native macOS and Windows app contexts add a second layer of serialization. Each host has a single app driver, and two native app contexts can't run against it at once without overwriting each other's session. So Doc Detective serializes native app contexts on that shared driver: a recording app context takes its turn alongside every other native app context on the host, whether or not that other context records. This keeps recording a native app surface safe under `concurrentRunners` greater than 1—the recording context queues for the driver instead of colliding with a concurrent app context. Contexts that don't drive a native app driver or the display, such as `httpRequest` and `runShell` steps, keep running in parallel.

<Note>
Desktop screen capture records whatever is visible in the captured region. If another window overlaps the app during the recording, it appears in the video—Doc Detective doesn't hide or minimize other windows. On a dedicated CI runner this rarely matters. On a shared desktop, keep the recorded app in the foreground.
</Note>

## Recording on mobile devices

On Android and iOS contexts, recordings capture the **device screen** through the device itself (Android's screen recorder, the iOS simulator's recorder) without touching the host display. This means:

- Omit `engine`. It doesn't apply, and Doc Detective skips an explicit `browser` or `ffmpeg` engine on a mobile context with guidance.
- Mobile recordings run under full concurrency. They never serialize with FFmpeg recordings or other browser tests, because they don't touch the shared display.
- The recording is the whole device screen; there's no window cropping to configure.
- All output formats work: Doc Detective saves `.mp4` directly and converts `.webm` and `.gif` after the recording stops.
- One recording can run per device at a time, and a single recording caps at 30 minutes.

```json
{
  "record": {
    "path": "android-flow.mp4",
    "surface": { "app": "settings" }
  }
}
```

A `record` step that runs before the first `startSurface` on a mobile context waits for the device session and starts recording as soon as the first app surface opens.

## Multiple overlapping recordings

A single context can run more than one recording at once. For example, you can capture a full test alongside a shorter clip of one workflow inside it. Give each `record` step a `name` so a later `stopRecord` step can target the right one:

- An untargeted `stopRecord` (`true` or `null`) stops the most recently started recording that's still active (last in, first out).
- To stop a specific recording, target it by `name`, either as a string (`stopRecord: "<name>"`) or an object (`stopRecord: { name: "<name>" }`). This works for any active recording, not just the most recent one.

Names must be unique among recordings that are active at the same time. If a `record` step reuses the `name` of a recording that's still active, Doc Detective catches the conflict before the run starts and skips that test in every context with a warning, rather than failing partway through.

When two recordings overlap and both would use the `browser` engine, the second one falls back to the `ffmpeg` engine with a `viewport` target and emits a warning, because the `browser` engine can own only one recorder per context. See [Recording engines](#recording-engines) for details on engine selection.

## Automatic recordings

To record every driver-based test context from start to finish—browser, native app, and mobile contexts alike—enable the `autoRecord` option. It captures each context end-to-end without explicit `record` steps, which is useful for:

- Building a video archive of every run
- Replaying exactly what happened when a test fails
- Comparing behavior from one run to the next

Enable it at the config, spec, or test level (test overrides spec, which overrides config):

```json
{
  "autoRecord": true,
  "output": "./test-results"
}
```

Or use the CLI flag:

```bash
doc-detective --auto-record
```

Use `--no-auto-record` to override a config file that enables it.

Automatic recordings wrap the whole context: recording starts before the first step and stops after the last. Desktop contexts capture with the `ffmpeg` engine. In an app-only context the capture starts full-display, and the first app surface to open scopes it to that app's window. Android and iOS contexts record the device screen instead, starting as soon as the first app surface opens its device session. Doc Detective saves each video in the per-run artifact folder (`<output>/.doc-detective/runs/<runId>/`) under the nested resource tree, so the same context lands on the same relative path in every run's folder:

```
.doc-detective/runs/<runId>/
└── specs/
    └── <specId>/
        └── tests/
            └── <testId>/
                └── contexts/
                    └── <contextId>/
                        └── recordings/
                            └── <contextId>.mp4
```

An automatic recording coexists with any explicit `record` and `stopRecord` steps in the same context. An untargeted `stopRecord` never stops the automatic recording, so a stray `stopRecord: true` can't cut the full-context video short. Doc Detective finalizes it during end-of-context cleanup.

<Note>
On Linux, each runner records its own virtual display with Xvfb, so automatic recordings stay isolated even when runs are concurrent. On Windows and macOS, concurrent automatic recordings share the same physical display and overlap, so Doc Detective warns when a run records concurrently on those platforms.
</Note>

## Recording checkpoints

A recording documents a flow, but the steps can keep passing while the recorded content quietly drifts—a redesigned page, a moved button, changed copy. To detect that drift, enable `checkpoints` on a `record` step:

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "checkpoints": true
  }
}
```

While the recording is active, Doc Detective captures a screenshot after every step (the `record` step's own capture is the opening bookend) and compares each against a persistent baseline stored beside the recording—`signup-flow.mp4.checkpoints/` by default. On the first run, the baselines don't exist yet, so `stopRecord` seeds them from the captures. Commit the baseline folder alongside your recording. On every later run, each checkpoint compares against its baseline:

- Variation within `maxVariation` (default `0.05`): the step passes.
- Variation beyond `maxVariation`: the `stopRecord` step returns WARNING—the recorded flow's content has probably changed since the recording was captured. Baselines are never modified by a checkpoint comparison.

The `stopRecord` step reports per-checkpoint results in its [outputs](stopRecord#outputs): `checkpoints` (file name and variation per checkpoint), `maxCheckpointVariation`, and `seededBaselines`.

Tune or relocate the comparison with the object form:

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "checkpoints": {
      "maxVariation": 0.1,
      "directory": "baselines/signup-flow"
    }
  }
}
```

A relative `directory` resolves beside the recording. Raise `maxVariation` (or disable checkpoints) for spans that show dynamic content—animations, ads, clocks—which otherwise produce WARNING noise. Checkpoints apply only to explicit `record` steps ([automatic recordings](#automatic-recordings) save to per-run folders, where baselines couldn't persist), and each checkpoint costs one extra screenshot per step while the span is active.

### Refreshing stale recordings automatically

Checkpoints detect drift; `overwrite: "aboveVariation"` acts on it. With this setting, the span always records, but the new video replaces the existing file **only when the span meaningfully changed**—checkpoint drift beyond `maxVariation`, an added, removed, or edited step, or a missing recording file. Otherwise Doc Detective discards the fresh capture and leaves the recording and its baselines byte-untouched, so unchanged recordings stay stable across runs:

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "overwrite": "aboveVariation"
  }
}
```

`aboveVariation` requires checkpoints, so it enables them automatically—including over an explicit `checkpoints: false`, which this mode overrides (a verdict with no evidence could never refresh a drifted recording). Add a `checkpoints` object to tune them. When the recording refreshes, its baselines update with it—the two always stay in sync—and the `stopRecord` step reports `changed: true` with the reasons in its outputs. If a step fails during the span, Doc Detective keeps the existing recording and baselines untouched rather than judging drift from an incomplete run.

### Verifying the video file itself

Checkpoints validate the scene a recording shows; `verify` validates the produced file. Use it to assert structural properties of the video—the classic capture failure modes are an all-black video, a wrong-sized crop, or a truncated clip:

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "verify": {
      "minDuration": 2,
      "resolution": true,
      "notBlack": true
    }
  }
}
```

Unlike checkpoint drift (a WARNING), a violated guard **fails** the `stopRecord` step—these are properties you explicitly demand of the artifact:

- `minDuration` / `maxDuration`: the video's length in seconds. Allow about half a second of slop for `.gif` targets, whose duration reporting is imprecise.
- `resolution`: the video's dimensions, with a ±2 pixel tolerance (encoders round to even dimensions). `true` compares against the capture plan's expectation—the crop rectangle for `window`/`viewport` targets, else the capture frame size (skipped when the engine reports no expectation). Pass `{ "width": …, "height": … }` to compare literal dimensions.
- `notBlack`: fails when black frames cover 95% or more of the video—the symptom of a broken screen capture.

The measured values land in the [`stopRecord` outputs](stopRecord#outputs) (`$$duration`, `$$width`, `$$height`, plus `$$resolutionMatch` and `$$allBlack` when the matching guard ran).

### Detecting staleness in headless runs

Recording needs a visible display, so headless runs skip `record` steps. With `aboveVariation` (or `checkpoints`) configured, a headless run still compares checkpoints against the committed baselines and reports whether the recording has gone stale: the `stopRecord` step returns WARNING with `stale: true` in its outputs, and a post-run hint suggests re-running headed to refresh. Your headless CI continuously watches recording freshness; you only run headed when something actually changed.

> For comprehensive options, see the [`record`](/reference/schemas/record) and [`stopRecord`](/reference/schemas/stoprecord) references.

## Examples

Here are a few ways you might use the `record` and `stopRecord` actions:

### Basic recording (boolean shorthand)

Starts recording to a default path like `output/record_1745032062266.mp4` and stops later.

{/* test {"testId":"record-boolean-shorthand","detectSteps":false} */}

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording.",
          "record": true
        },
        {
          "description": "Perform some actions...",
          "goTo": "https://example.com"
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```

{/* step {"stepId":"record-start-boolean","description":"Start recording, directed at a gitignored output folder instead of the default path (on this headless docs-test runner, with no virtual display, this step SKIPS rather than fails -- see PR notes)","record":{"path":"record_boolean.mp4","directory":"./.dd-output"}} */}
{/* step {"stepId":"record-action-boolean","description":"Perform some actions","goTo":"http://localhost:8092"} */}
{/* step {"stepId":"record-stop-boolean","description":"Stop recording","stopRecord":true} */}
{/* test end */}

### Recording to a specific path (string shorthand)

Records to `./output/media/login_flow.webm`.

{/* test {"testId":"record-string-shorthand","detectSteps":false} */}

```json
      "steps": [

          "description": "Stop recording.",

        }

    }
  ]
}
```
