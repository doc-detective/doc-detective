title: record
description: Start capturing a video of test execution.

The `record` action starts recording test execution as a video file (MP4, WebM, or GIF). This is useful for debugging failed tests or creating multimedia documentation. Recordings are video only, with no audio track. Recording continues until a `stopRecord` step is encountered.

You can specify the recording action in several ways:

- **Boolean Shorthand:** Set `record: true` to start recording with default settings (usually saved as `record_<timestamp>.mp4` in the output directory).
- **String Shorthand:** Provide a file path directly as the value for the `record` key (for example, `record: "test_session.webm"`). This starts recording to the specified path. Supported extensions are `.mp4`, `.webm`, and `.gif`.
- **Object Format:** Use an object for more control over the output file:
  - `path`: (Optional) The file path where the recording will be saved. Can include directories. If omitted, a default path is generated.
  - `directory`: (Optional) The directory where the recording will be saved. If `path` includes a directory, this is ignored. Defaults to the configured output directory.
  - `overwrite`: (Optional) Set to `true` to overwrite an existing file at the target path. Defaults to `false`.
  - `engine`: (Optional) The recording engine to use. See [Recording engines](#recording-engines) below.

**Stopping the Recording:** You *must* include a `stopRecord` step later in your test to finalize and save the video file.

## Recording engines

Recording uses one of two engines, selected by the optional `engine` field in the object format:

- **`browser`:** Records the Chrome viewport using the browser's screen-capture API. It captures only the browser tab and is safe to run with concurrency (when `concurrentRunners` is greater than 1), because each context records its own window. The `browser` engine requires a visible Chrome context and can't record in headless mode.
- **`ffmpeg`:** Records the screen with ffmpeg, so it works for any application, not just Chrome. Doc Detective installs ffmpeg automatically as a runtime dependency the first time a recording runs, so you don't need to install it yourself.

If you don't set `engine`, Doc Detective chooses one for you. It prefers the `browser` engine when a visible Chrome context is available (coercing an unspecified browser to headed Chrome), and falls back to `ffmpeg` otherwise.

Set `engine` as a string to select an engine with its defaults (for example, `"engine": "ffmpeg"`), or as an object for more control:

- `name`: (Required) The engine to use, either `browser` or `ffmpeg`.
- `target`: (Optional, `ffmpeg` only) What the engine captures. Use `display` to record the full screen, `window` to record the active window, or `viewport` to record the browser content area (default: `display`). The `window` and `viewport` targets are best-effort: ffmpeg captures the full screen and then crops the result. The `browser` engine ignores this field and always captures its tab.
- `fps`: (Optional, `ffmpeg` only) The capture frame rate (default: `30`).

**Recording with concurrency:** Concurrency support depends on the engine. The `browser` engine is always safe to run concurrently, because each context records its own window.

The `ffmpeg` engine captures the whole shared display, so two `ffmpeg` recordingsâ€”or an `ffmpeg` recording alongside another browser-driven testâ€”would overwrite each other's output. To prevent this, whenever a run includes an `ffmpeg` recording, Doc Detective serializes that recording with every other browser-based test so they take turns on the display. Jobs that don't drive a browser, such as `httpRequest` and `runShell` steps, keep running in parallel up to `concurrentRunners`. This behavior applies on every platform, including Linux. To run the recordings concurrently as well, use the `browser` engine instead.

On Linux, the `ffmpeg` engine also needs Xvfb to record headless contexts, which have no physical display to capture. Install Xvfb to record in headless mode.

> For comprehensive options, see the [`record`](/reference/schemas/record) and [`stopRecord`](/reference/schemas/stoprecord) references.

## Examples

Here are a few ways you might use the `record` and `stopRecord` actions:

### Basic recording (boolean shorthand)

Starts recording to a default path like `output/record_1745032062266.mp4` and stops later.

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording.",
          "record": true
        },
        {
          "description": "Perform some actions...",
          "goTo": "https://example.com"
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```

### Recording to a specific path (string shorthand)

Records to `./output/media/login_flow.webm`.

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording to a specific WebM file.",
          "record": "./output/media/login_flow.webm"
        },
        {
          "description": "...",
          "find": "Login"
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```

### Recording with object format (overwrite enabled)

Records to `test_video.gif` in the `output/gifs` directory, overwriting if it exists.

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording a GIF, overwriting if necessary.",
          "record": {
            "path": "test_video.gif",
            "directory": "./output/gifs",
            "overwrite": true
          }
        },
        {
          "description": "...",
          "wait": 2000
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```

### Recording any application with the ffmpeg engine

Records the screen with ffmpeg so the recording isn't limited to Chrome.

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording the screen with ffmpeg.",
          "record": {
            "path": "demo.mp4",
            "engine": "ffmpeg"
          }
        },
        {
          "description": "...",
          "runShell": "my-cli --help"
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```

### Configuring the ffmpeg engine (detailed object)

Records the active window at 60 fps with ffmpeg.

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Start recording the active window.",
          "record": {
            "path": "demo.mp4",
            "engine": {
              "name": "ffmpeg",
              "target": "window",
              "fps": 60
            }
          }
        },
        {
          "description": "...",
          "wait": 2000
        },
        {
          "description": "Stop recording.",
          "stopRecord": true
        }
      ]
    }
  ]
}
```
