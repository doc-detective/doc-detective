title: click

The `click` action allows you to click or tap an element on the page. You can specify which mouse button to use and target elements using text or selectors.

The `click` action works in several ways:

- **String Shorthand:** The display text or selector of the element to find and click.
- **Object Format:** Use an object with detailed properties for more control over the click action.

If you need to find an element before clicking it, consider using the [`find`](/docs/actions/find) action, which lets you locate elements and optionally click them in a single step.

**Note:** If you use the `click` action without specifying a target (using element text or selector), it clicks the active element or the element at the current cursor position.

## Properties

When using the object format, you can specify:

- `button`: (Optional) Kind of click to perform. Can be `"left"`, `"right"`, or `"middle"`. Default is `"left"`.
- `duration`: (Optional) How long to hold the press, in milliseconds. A long-press (touch-and-hold) on mobile app surfaces, press-and-hold of the primary button on desktop apps, and press-and-hold of the given `button` on browsers. Omit for a normal click. A long-press on an app surface is primary-button only, so `duration` with a non-left `button` fails the step. On browsers, a held click needs the W3C actions endpoint, so—like non-left buttons—it's desktop-browser-only.
- `selector`: (Optional) [CSS selector](/docs/selectors/css) or [XPath selector](/docs/selectors/xpath) of the element to click. If combined with `elementText`, the element must match both the text and the selector.
- `elementText`: (Optional) Display text of the element to click. If combined with `selector`, the element must match both the text and the selector.
- `surface`: (Optional) The browser (session, window, or tab) the element lives in, for example `{ "browser": "chrome", "tab": "cart" }` or a named browser `{ "browser": "firefox", "name": "admin" }`, or the [native app surface](/docs/actions/startsurface) it lives in, for example `{ "app": "notepad" }`. Omit to act on the active surface (the most recently opened, focused, or explicitly targeted surface, whether that's a browser tab or an app window). The targeted surface stays active afterward, so later surface-less steps act on it too. On an app surface, `elementText`, `elementId`, and `elementAria` map to native controls, and `selector` accepts a native locator (XPath or a `~`-prefixed accessibility ID) instead of a CSS selector. See [Test across multiple tabs, windows, and browsers](/docs/test-docs/multiple-tabs-and-windows) and [Target an app surface](/docs/actions/startsurface#target-an-app-surface).

**Outputs:** when you target an element by text or selector, the action exposes `outputs.found`, a boolean indicating whether the element was located. To capture it for later steps, use the step-level `variables` object with the expression `$$found`.

> For comprehensive options, see the full [`click`](/reference/schemas/click) reference.

## Examples

Here are a few ways you might use the `click` action:

### Click by element text (string shorthand)

{/* test {"testId":"click-by-text","detectSteps":false} */}

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Click on an element with text 'Submit'",
          "click": "Submit"
        }
      ]
    }
  ]
}
```

{/* step {"stepId":"goto-click-text","description":"Open the click test page","goTo":"http://localhost:8092/click-shorthand.html"} */}
{/* step {"stepId":"click-text","description":"Click the button by its display text","click":"Reveal by text"} */}
{/* step {"stepId":"verify-click-text","description":"The button appends a message only when actually clicked","find":"Text click landed"} */}
{/* test end */}

### Click by selector (string shorthand)

{/* test {"testId":"click-by-selector","detectSteps":false} */}

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Click on an element matching the CSS selector",
          "click": "#submit-button"
        }
      ]
    }
  ]
}
```

{/* step {"stepId":"goto-click-selector","description":"Open the click test page","goTo":"http://localhost:8092/click-shorthand.html"} */}
{/* step {"stepId":"click-selector","description":"Click the button by its CSS selector","click":"#reveal-by-selector"} */}
{/* step {"stepId":"verify-click-selector","description":"Confirm the selector click fired","find":"Selector click landed"} */}
{/* test end */}

### Click an element by text (object format)

{/* test {"testId":"click-by-object","detectSteps":false} */}

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Left-click on a button with specific text",
          "click": {
            "elementText": "Submit",
            "button": "left"
          }
        }
      ]
    }
  ]
}
```

{/* step {"stepId":"goto-click-object","description":"Open the click test page","goTo":"http://localhost:8092/click-shorthand.html"} */}
{/* step {"stepId":"click-object","description":"Left-click the button using object format","click":{"elementText":"Reveal by object","button":"left"}} */}
{/* step {"stepId":"verify-click-object","description":"Confirm the object-format click fired","find":"Object click landed"} */}
{/* test end */}

### Click an element by selector (object format)

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Middle-click on an element with a specific selector",
          "click": {
            "selector": "#open-in-new-tab",
            "button": "middle"
          }
        }
      ]
    }
  ]
}
```

### Click an element by both selector and text

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Click on a specific element matching both selector and text",
          "click": {
            "selector": ".btn",
            "elementText": "Download",
            "button": "left"
          }
        }
      ]
    }
  ]
}
```

### Long-press an element on a mobile app surface

```json
{
  "tests": [
    {
      "steps": [
        {
          "description": "Touch and hold a message for 800ms",
          "click": {
            "elementText": "Message",
            "duration": 800,
            "surface": { "app": "myapp" }
          }
        }
      ]
    }
  ]
}
```

## Mobile app surfaces

On an app surface, `click` is a tap, and `duration` makes it a long-press. The `button` field (`left`, `right`, or `middle`) works where the platform's driver supports it. Windows honors all three. macOS honors right-click and rejects middle-click, which the Mac2 driver can't do. Android and iOS reject any non-left button with a clear message, because touch input has no right or middle button. On Windows, plain left clicks activate the element through the UI Automation Invoke pattern (coordinate-free, so display scaling can't misdirect them) and fall back to a physical click for elements that don't support it. Non-left and held clicks stay physical. To click in a specific window of the app, add a `window` selector (see [App windows](find#app-windows)). Two more patterns worth knowing:

- **Permission dialogs are just elements.** When your documentation walks the reader through granting a permission, click the dialog's button like any other element: `{ "click": { "elementText": "Allow", "surface": { "app": "myapp" } } }`. System dialogs expose their buttons through the platform accessibility tree, so `elementText` matches them.
- **When the dialog isn't the point**, suppress it at session start instead of scripting around it: pass the driver's own knob through `startSurface.driverOptions`, for example `{ "driverOptions": { "appium:autoGrantPermissions": true } }` on Android. Use this when the procedure you're testing starts *after* the reader grants permissions.
