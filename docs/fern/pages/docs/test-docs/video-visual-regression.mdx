description: Use checkpoints and verify guards to catch visual drift and structural problems in a recording.

A recording's steps can keep passing while the content quietly drifts. Enable checkpoints to compare each step's frame against a committed baseline, and add `verify` guards to assert structural properties of the produced video file. New to recording? Start with [Record video](/docs/test-docs/record-video-guide).

## Before you begin

Make sure you have the following:

- [Doc Detective installed](/docs/get-started/installation).
- A directory where you can create a `.spec.json` file.
- A headed, visible context to capture a *fresh* recording—checkpoints and `verify` guards need a real recording from a visible display. Staleness detection against committed baselines still runs headless (see [Detect staleness without re-recording](#detect-staleness-without-re-recording)).
- A committed baseline folder alongside the recording (the first run seeds it).

The steps in this guide work the same whether you author them in a standalone `.spec.json`, [inline in a doc source file](/docs/test-docs/inline), or let Doc Detective [detect them from prose](/docs/test-docs/detected).

## Detect visual regressions with checkpoints

A recording's steps can keep passing while the recorded content quietly drifts—a redesigned page, a moved button, changed copy. To catch that, enable `checkpoints` on a `record` step. Doc Detective then captures a screenshot after every step during the span and compares each against a committed baseline stored beside the recording (`signup-flow.mp4.checkpoints/` by default, or an object-form `directory`):

{/* test {"testId":"record-guide-checkpoints","detectSteps":false,"runOn":[{"platforms":["windows","mac","linux"],"browsers":{"name":"chrome","headless":false}}]} */}

```json
{
  "record": { "path": "signup-flow.mp4", "checkpoints": true }
}
```

{/* step {"stepId":"record-guide-cp-goto","description":"Open the local test server","goTo":"http://localhost:8092"} */}
{/* step {"stepId":"record-guide-cp-start","description":"Start a checkpointed recording, directed at a gitignored output folder; this test is gated to headed Chrome via runOn because checkpoints need a real recording from a visible display, so it is SKIPPED on the headless docs-test runner and seeds baselines on a headed run","record":{"path":"guide_checkpoints.mp4","directory":"./.dd-output","checkpoints":true}} */}
{/* step {"stepId":"record-guide-cp-step","description":"Take a step during the span","goTo":"http://localhost:8092/enhanced-elements.html"} */}
{/* step {"stepId":"record-guide-cp-stop","description":"Stop recording","stopRecord":true} */}
{/* test end */}

On the first run the baselines don't exist yet, so `stopRecord` seeds them from the captures. Commit the baseline folder alongside the recording. On every later run, each checkpoint compares against its baseline:

- Within `maxVariation` (default `0.05`): the step passes.
- Beyond `maxVariation`: `stopRecord` returns `WARNING`, since the recorded flow's content has probably changed. A comparison never modifies the baselines.

Tune or relocate the comparison with the object form, and raise `maxVariation` or disable checkpoints for spans that show dynamic content—animations, ads, clocks—which otherwise produce `WARNING` noise:

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "checkpoints": { "maxVariation": 0.1, "directory": "baselines/signup-flow" }
  }
}
```

To act on drift instead of only reporting it, set `overwrite: "aboveVariation"`. The span always records, but the new video replaces the existing file only when the span meaningfully changed—checkpoint drift beyond `maxVariation`, an added, removed, or edited step, or a missing recording file. Otherwise Doc Detective discards the fresh capture and leaves the file and baselines untouched, so unchanged recordings stay stable across runs. This mode implies checkpoints and, when it refreshes, updates the baselines with the video in the same step:

```json
{
  "record": { "path": "signup-flow.mp4", "overwrite": "aboveVariation" }
}
```

### Verify the video file itself

Checkpoints validate the scene a recording shows; `verify` validates the produced file. Use it to assert structural properties of the video—the classic capture failure modes are a wrong-sized crop, a truncated clip, and an all-black video:

{/* test {"testId":"record-guide-verify","detectSteps":false,"runOn":[{"platforms":["windows","mac","linux"],"browsers":{"name":"chrome","headless":false}}]} */}

```json
{
  "record": {
    "path": "signup-flow.mp4",
    "verify": { "minDuration": 2, "resolution": true, "notBlack": true }
  }
}
```

{/* step {"stepId":"record-guide-verify-goto","description":"Open the local test server","goTo":"http://localhost:8092"} */}
{/* step {"stepId":"record-guide-verify-start","description":"Start a recording with structural verify guards, directed at a gitignored output folder. Gated to headed Chrome via runOn because the verify guards (minDuration, resolution, notBlack) can only hold against a real recording from a visible display; it SKIPS on any platform whose only Chrome is headless and exercises the guards on the headed runner","record":{"path":"guide_verify.mp4","directory":"./.dd-output","verify":{"minDuration":2,"resolution":true,"notBlack":true}}} */}
{/* step {"stepId":"record-guide-verify-wait","description":"Let the recording run past the minDuration guard (2 s) before stopping","wait":2500} */}
{/* step {"stepId":"record-guide-verify-stop","description":"Stop recording","stopRecord":true} */}
{/* test end */}

Unlike checkpoint drift, which returns `WARNING`, a violated `verify` guard fails the `stopRecord` step—these are properties you explicitly demand of the artifact:

- `minDuration` / `maxDuration`: the video's length in seconds.
- `resolution`: the video's dimensions, with a ±2 pixel tolerance because encoders round to even dimensions. `true` compares against the capture plan's expectation, or pass `{ "width": …, "height": … }` for literal dimensions.
- `notBlack`: fails when black frames cover 95% or more of the video—the symptom of a broken screen capture.

### Detect staleness without re-recording

Doc Detective skips the capture in two situations: a headless run, which has no visible display, and a target file that already exists under the default `overwrite: "false"`. In both, a span with `checkpoints` (or `aboveVariation`, which implies them) still compares its checkpoints against the committed baselines and reports whether the recording has gone stale. `stopRecord` returns `WARNING` with `stale: true`, and a post-run hint points at the fix, which depends on the skip:

- **Headless run:** re-run the test on a headed context to refresh the recording.
- **Target already exists:** set `overwrite` to `"true"` or `"aboveVariation"`, or remove the file.

Detection is cheap and continuous, so headless CI watches recording freshness on every run and you only re-record when something actually changed. Skipped spans never write: no video, no baseline seeding, no baseline updates.

`stopRecord` reports the outcome of all of this in its outputs: `$$checkpoints` and `$$maxCheckpointVariation`, `$$seededBaselines`, `$$changed` and `$$changeReasons`, `$$stale`, `$$duration` / `$$width` / `$$height`, and `$$resolutionMatch` / `$$allBlack`. For the full list, see [Recording checkpoints](/docs/actions/record#recording-checkpoints), the [`stopRecord`](/docs/actions/stoprecord) action reference, and the [`stopRecord`](/reference/schemas/stoprecord) schema.

## Verify it works

- Checkpoint drift beyond `maxVariation` reports `WARNING`, so the run continues while flagging that the recorded content probably changed.
- A violated `verify` guard reports `FAIL`, because you demanded that structural property of the file.
- A skipped capture whose baselines diverged reports `WARNING` with `stale: true`.

## Troubleshooting

### `stopRecord` returns WARNING with `stale: true`

**Cause:** the capture was skipped—a headless run or an existing file—while the committed baselines diverged from the current content.

**Solution:** re-run on a headed context to refresh the recording, or set `overwrite` to `"true"` or `"aboveVariation"`.

### A `record` step is skipped and the file isn't refreshed

**Cause:** the default `overwrite: "false"` skips a `record` step whose file already exists.

**Solution:** set `overwrite` to `"true"` to always re-record, or `"aboveVariation"` to re-record only when the span meaningfully changed.
