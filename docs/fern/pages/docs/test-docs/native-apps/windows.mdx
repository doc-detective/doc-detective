Doc Detective can open and drive native Windows desktop apps as part of a test. This experimental workflow launches an app with [`startSurface`](/docs/actions/startsurface), acts on it with the same `find`, `click`, and `type` steps you already use in the browser, and closes it with [`closeSurface`](/docs/actions/closesurface).

<Info>

**Experimental**: Native and mobile app testing is experimental and subject to change.

</Info>

## Before you begin

Windows app tests run only on a **Windows host**. On any other platform, Doc Detective skips the test with a reason rather than failing it, so a mixed-platform spec stays green.

Doc Detective drives Windows apps through the `appium-novawindows-driver` (NovaWindows) automation driver. It installs the driver automatically the first time a Windows app surface starts, so you don't have to prepare anything for a first run. There's no separate pre-install command for this driver: it only installs lazily on that first launch. See the [`install` command](/reference/cli/install) reference for the runtime packages that do support pre-installation.

## Run a first test

This test targets the `windows` platform, launches Character Map by its executable path, waits until the window is interactive, clicks a control, and closes the app. Character Map ships with Windows, so the test needs nothing else installed.

```json
{
  "tests": [
    {
      "runOn": [
        {
          "platforms": ["windows"]
        }
      ],
      "steps": [
        {
          "description": "Launch the app and wait until it's ready.",
          "startSurface": {
            "app": "C:\\Windows\\System32\\charmap.exe",
            "timeout": 120000,
            "waitUntil": {
              "find": { "elementText": "Select" }
            }
          }
        },
        {
          "description": "Click a control.",
          "click": {
            "elementText": "Select",
            "surface": { "app": "charmap" }
          }
        },
        {
          "description": "Capture the window.",
          "screenshot": {
            "path": "charmap.png",
            "surface": { "app": "charmap" }
          }
        },
        {
          "description": "Close the app.",
          "closeSurface": { "app": "charmap" }
        }
      ]
    }
  ]
}
```

To run this test, save it to a spec file and pass it to Doc Detective:

```bash
doc-detective --input windows-app.spec.json
```

A few things to notice:

- The generous `timeout` (120,000 ms) leaves room for the driver to install on a first run and for the app to launch.
- `waitUntil.find` gates readiness: the step doesn't finish until an element with the text `Select` exists, so the next step never runs against a window that hasn't drawn its controls yet.
- Later steps address the app with `surface: { "app": "charmap" }`. Doc Detective derives the default name `charmap` from the executable file name; pass a `name` on `startSurface` to choose your own.
- The [`screenshot`](/docs/actions/screenshot) step captures a fresh image of the app every run, so the picture never drifts out of sync with the current build.

To launch a packaged (UWP) app instead of an executable, set `app` to its AppUserModelID, such as `"Microsoft.WindowsCalculator_8wekyb3d8bbwe!App"`. See the [app descriptor](/reference/schemas/app-descriptor) reference for the identifier syntax.

## Target only Windows

Because an app targets one platform, gate the test with a [context](/docs/test-docs/platforms-and-browsers) so it runs only where it can:

```json
{
  "runOn": [
    {
      "platforms": ["windows"]
    }
  ]
}
```

On a host that isn't Windows, the test skips with the reason that it targets a different platform. See [Test across platforms and browsers](/docs/test-docs/platforms-and-browsers) for the full context grammar.

## Troubleshooting

**Element not found**

- **Cause:** Windows exposes controls through UI Automation, so element criteria map onto UIA properties: `elementText` matches a control's Name, `elementId` and `elementTestId` match its AutomationId, and `elementAria` matches its control type. A criterion that doesn't line up with those properties won't match.
- **Solution:** Match against the UIA property that carries the value you're looking for. Desktop finds don't auto-scroll, because UI Automation exposes off-screen controls directly; a control that isn't visible is still findable, but one that genuinely isn't in the tree won't match. See [Target an app surface](/docs/actions/startsurface#target-an-app-surface) and the [`find`](/docs/actions/find) reference.

**Launch arguments don't reach the app**

- **Cause:** On Windows, the driver takes launch arguments as a single command string. An `args` entry that contains spaces gets split unless it carries its own quotes.
- **Solution:** Join your arguments into one string and quote any argument that contains spaces. See the [app descriptor](/reference/schemas/app-descriptor) reference.

**The wrong window is targeted**

- **Cause:** An app with more than one window defaults to its main window.
- **Solution:** Add a `window` selector to the `surface` (an index, where `-1` is the newest, a title, or an object form) to address a specific window. See [App windows](/docs/actions/find#app-windows).

**A working directory setting is ignored elsewhere**

- **Cause:** `workingDirectory` is supported on Windows only. On other platforms, setting it fails the step.
- **Solution:** Keep `workingDirectory` in Windows-only specs. See the [app descriptor](/reference/schemas/app-descriptor) reference.
