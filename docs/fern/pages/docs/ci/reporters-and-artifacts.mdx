Every Doc Detective run produces output: a summary in your terminal, machine-readable results, and—when your tests capture them—screenshots and recordings. This guide shows you how to choose which reporters run, where each one writes its output, and how to fail a CI build when tests fail.

For the full list of reporter and output options, see the [`config`](/reference/schemas/config) reference.

## Before you begin

- [Install Doc Detective](/docs/get-started/installation) and have a spec you can run.
- Know your output directory. Reporters that write files use the `output` config option (or the `--output` flag); it defaults to the current working directory.

## Step 1: Choose your reporters

Doc Detective ships four reporters: `terminal`, `json`, `html`, and `runFolder`. By default, it runs `terminal`, `json`, and `runFolder`. To choose a different combination, set the `reporters` option.

| Reporter | Output |
|---|---|
| `terminal` | A colored pass/fail/warning/skipped summary printed to stdout. Writes no files. |
| `json` | A flat JSON results file with the full step-by-step outcome. |
| `html` | A self-contained HTML report you can open in a browser. |
| `runFolder` | A per-run folder containing the results (JSON and HTML) alongside the screenshots and recordings the run captured. |

Select reporters on the command line:

```bash
doc-detective --input api.spec.json --reporters terminal json html
```

Or set them in your config file so every run uses the same combination:

```json title=".doc-detective.json"
{
  "reporters": ["terminal", "json", "html"]
}
```

## Step 2: Read the terminal summary

The `terminal` reporter prints a summary grouped by specs, tests, contexts, and steps, with counts for passed, failed, warnings, and skipped. When everything passes and at least one test ran, it prints a success banner. When no tests run—because of an empty input or an active filter—it prints a warning instead of a success message, so an empty run never looks like a green run.

## Step 3: Find the JSON results

The `json` reporter writes the full results to a file:

- If `output` ends in `.json`, Doc Detective writes to that path, adding a numeric suffix (`-0.json`, `-1.json`, …) to avoid overwriting an existing file.
- Otherwise it writes `testResults-<timestamp>.json` in the output directory.

The results file carries a `summary` object with the pass, fail, warning, and skipped counts, followed by the per-spec, per-test, per-context, and per-step detail. For an example of the results structure, see [Create your first test](/docs/get-started/create-your-first-test#outcome). To act on results programmatically, parse this file (see [Fail CI when tests fail](#step-6-fail-ci-when-tests-fail)).

### Read the warm-phase timings

The results also carry a `warm` object that records what the run provisioned before executing tests. At the start of every run, Doc Detective front-loads the setup its tests would otherwise pay one at a time—browser and driver installs, simulator and emulator boots, and related downloads—and runs those tasks concurrently. Each entry lists the task, its outcome, and how long it took:

```json
"warm": {
  "durationMs": 4211,
  "tasks": [
    { "name": "browser-install:chrome", "kind": "browser-install", "outcome": "skipped", "durationMs": 2, "note": "'chrome' is already available" },
    { "name": "session-probe", "kind": "session-probe", "outcome": "warmed", "durationMs": 4205, "note": "1 combination ok" }
  ]
}
```

Outcomes are `warmed` (Doc Detective performed the work), `skipped` (nothing to do, or the environment handles it later), and `failed`. The warm phase is best-effort: a `failed` task logs a warning and never fails the run—the affected test context retries or skips with its normal behavior and reason. Use the timings to spot provisioning costs in slow CI runs: a long `warmed` task on every run usually means the [runtime cache](/docs/ci/docker-and-headless#step-3-warm-runtime-assets) isn't persisted between runs.

## Step 4: Find the HTML report

The `html` reporter writes a self-contained HTML report you can open in a browser:

- If `output` ends in `.html` or `.htm`, Doc Detective writes to that path, adding a numeric suffix (`-0.html`, `-1.html`, …) to avoid overwriting an existing file.
- Otherwise it treats `output` as a directory and writes `testResults-<timestamp>.html` there (or `coverageResults-<timestamp>.html` for a coverage run).

The report embeds the same results the `json` reporter produces, so a single file is enough to share a run with someone who doesn't have Doc Detective installed.

## Step 5: Collect the run folder and artifacts

The `runFolder` reporter archives each run into its own folder so you can attach it to a CI job or diff one run against another:

```
<output>/.doc-detective/runs/<runId>/
├── testResults.json
├── testResults.html
└── specs/<specId>/
    └── tests/<testId>/
        └── contexts/<contextId>/
            ├── screenshots/<NN>-<action>-<stepId>.png
            └── recordings/<contextId>.mp4
```

The file names inside the folder are constant—the run ID lives in the folder name—so two run folders compare cleanly. Auto-captured screenshots and recordings land beside the results in the same folder, which makes the whole folder a single artifact to upload from CI for reviewers to inspect.

## Step 6: Fail CI when tests fail

This is the most common mistake when wiring Doc Detective into a pipeline, so read it carefully.

<Warning>
By default, the standalone `doc-detective` CLI does **not** exit with a non-zero code when tests fail. It exits non-zero only when it crashes or your config is invalid. A pipeline that relies on the CLI's exit code reports success even when documentation tests failed—unless you opt in with `--exit-on-fail`.
</Warning>

To gate your build, choose one of these approaches:

- **Set `--exit-on-fail` (simplest, any CI platform).** Pass the flag (or set `exitOnFail: true` in your config) and the CLI exits with code `1` whenever a spec fails, so the job fails like a broken unit test. This is the least setup and works the same everywhere—GitLab CI, Jenkins, CircleCI, or a bare shell.

```bash
doc-detective --input ./docs --exit-on-fail
```

- **Use the GitHub Action.** Set its `exit_on_fail` input to `true` so the job fails when tests fail. See the [GitHub Action guide](/docs/ci/github-action).
- **Parse the JSON results.** Read the `summary` object from the `json` reporter's output and fail the step yourself when any `fail` count is greater than zero. Useful when you want custom logic beyond a simple pass/fail gate.

```bash
doc-detective --input ./docs --reporters json --output results.json
node -e "process.exit(require('./results.json').summary.tests.fail > 0 ? 1 : 0)"
```

`--exit-on-fail` gates on failing specs; `WARNING` results are non-fatal and never trigger a non-zero exit.

## Verify it works

After a run, confirm that:

- The terminal shows the summary you expect.
- The `json` reporter wrote a results file at the path you configured.
- If you enabled `runFolder`, a `.doc-detective/runs/<runId>/` folder exists with `testResults.json` and any captured screenshots or recordings.
