---
title: Secrets
description: Reference credentials in tests without leaking them into reports and logs.
---

Tests often need a credential: an API token for a request, or a password to type into a login form. If you reference one as an ordinary variable, such as `$API_KEY`, Doc Detective resolves it and the resolved value appears in your test results, your logs, and your HTML report—anywhere you might paste or publish them.

Reference it as `$secret.API_KEY` instead. Doc Detective reads the value from the same environment variable, sends it to whatever you're testing, and keeps it out of everything it writes.

## Reference a secret

Use `$secret.` followed by the name of the environment variable:

```json
{
  "httpRequest": {
    "url": "https://api.example.com/data",
    "request": {
      "headers": { "Authorization": "Bearer $secret.API_KEY" }
    }
  }
}
```

The value comes from the environment, so anything that sets an environment variable works: your shell, the secret store in your CI platform, or a `.env` file loaded with [`loadVariables`](/docs/actions/loadVariables). Nothing else changes about how you supply credentials.

Use a secret anywhere the value goes to the thing you're testing:

- Request URLs, headers, parameters, and bodies
- Keys you `type` into a page
- Shell commands and code you run

A reference can sit inside a larger value, such as `"Bearer $secret.API_KEY"`, and you can use more than one in a step.

## What your reports show

The report keeps the reference rather than the value:

```json
{ "type": "$secret.LOGIN_PASSWORD", "result": "PASS" }
```

This tells you which credential the step used, which a masked blob can't.

Sometimes a secret comes back out of the thing you're testing: an API echoes your token, or a command prints its own arguments. Doc Detective replaces the value with `***secret.API_KEY***` wherever it appears. Don't match on that text in a test. It isn't a stable value.

## You can't assert on a secret

Doc Detective fails a step that uses a secret in a field it would have to compare or report:

- Expected values, such as `elementText`, `statusCodes`, or an `httpRequest` `response`
- Conditions in `if`, `assertions`, and routing such as `onFail`
- `variables` and `outputs`
- The `description` of a step

This limit is deliberate. A test that checks whether a secret matches something reveals the secret a little at a time, through the pass or fail, the retry count, and the message in the report. Assert on what the credential achieves instead: the status code it returns, or the element that appears only after you sign in.

```json
// Fails: compares the secret.
{ "find": { "elementText": "$secret.LOGIN_PASSWORD" } }

// Works: checks what a successful sign-in produced.
{ "find": { "elementText": "Sign out" } }
```

## When a secret isn't set

The step fails and names the variable:

```
Couldn't resolve secret `$secret.API_KEY`: the environment variable is unset or empty.
```

To skip a test when a credential isn't available, such as a suite that contributors run without your production tokens, gate the context with `requires`:

```json
{ "runOn": [{ "requires": { "env": ["API_KEY"] } }] }
```

The test then skips instead of failing.

A secret shorter than four characters also fails. Doc Detective can't mask a value that short—doing so would corrupt unrelated output—so rather than send a credential it can't keep out of your reports, it stops and tells you. Use a longer value.

## Credentials you didn't declare

Doc Detective also redacts values that *look* like credentials, even when you never referenced them with `$secret.`. It recognizes JSON Web Tokens, GitHub tokens, AWS access key IDs, and passwords embedded in connection strings such as `postgres://app:hunter2@db/prod`.

This is a safety net, not protection you can plan around. It matches shapes, so it can't recognize an ordinary password. Use `$secret.` for anything you rely on.

If a legitimate value in your output looks like a credential and gets redacted, turn the net off:

```json title=".doc-detective.json"
{ "heuristicRedaction": false }
```

Values you reference with `$secret.` stay protected either way. For details, see the [`heuristicRedaction`](/reference/schemas/config) reference.

## What secrets don't cover

Secret references protect what Doc Detective writes. They don't reach these:

- **Screenshots and recordings.** A credential typed into a visible field appears in the pixels. Skip recording the sign-in, or mask the field in your app.
- **Cookie files.** [`saveCookie`](/docs/actions/saveCookie) writes session cookies to disk as-is.
- **Values your commands change.** Masking matches the value, so a command that hashes or rewrites a secret and prints the result defeats it.

<Info>
Whatever you test sees a secret just like any other value. A shell command runs with the environment of your test run, so a script you call can read any variable in it.
</Info>
