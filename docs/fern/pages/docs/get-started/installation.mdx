Downloading and running Doc Detective is straightforward. You can install Doc Detective using NPX (Node.js) or Docker. Follow the instructions for your preferred method:

Choose *NPX* if you have Node.js installed and want to install Doc Detective on your machine locally. Choose *Docker* if you can't install applications locally, prefer containerized environments, need to avoid Node.js installation, or want to isolate Doc Detective from your system.

## Runtime installation

Running `npm i doc-detective` installs the CLI and pre-installs heavy runtime assets—browsers (Chrome, Firefox), drivers (ChromeDriver, Geckodriver), ffmpeg, and the npm packages that drive them (webdriverio, appium, sharp, etc.)—into `<os.tmpdir()>/doc-detective/` (or `DOC_DETECTIVE_CACHE_DIR`). This keeps your install ready to run tests without a separate step.

The pre-install runs in a child process and captures its output, so npm's deprecation warnings from transitive dependencies won't clutter your terminal.

### Opt out of pre-installation

To skip the heavy asset pre-install, set `DOC_DETECTIVE_AUTOINSTALL=0` (also accepts `false`, `no`, or `off`):

```bash
DOC_DETECTIVE_AUTOINSTALL=0 npm i doc-detective
```

With this option, the CLI install stays small. Heavy assets install lazily the first time a test needs them.

### Pre-install manually

If you opted out or need to reinstall, pre-install all runtime assets:

```bash
doc-detective install all --yes
```

### Install specific components

Install only what your tests need:

```bash
doc-detective install browsers chrome chromedriver
doc-detective install runtime webdriverio appium
doc-detective install agents
```

### Check installation status

See what's installed versus what the current version expects:

```bash
doc-detective install status
```

### Override the cache location

The default cache location (`<os.tmpdir()>/doc-detective/`) may not persist across reboots on some systems. For containers, CI runners, or any environment where you want the cache to survive, set a custom location:

```bash
DOC_DETECTIVE_CACHE_DIR=/opt/doc-detective doc-detective install all --yes
```

You can also set `cacheDir` in your `.doc-detective.json` config file or use the `--cache-dir` CLI flag.

### Prebuilt runtime cache

This is automatic and needs no configuration. Skip this section unless you run in an air-gapped or mirrored environment, or you want to tune how Doc Detective restores the cache.

To make the *first* run on a fresh machine fast, every Doc Detective release attaches prebuilt, per-platform tarballs of the warm cache—the heavy npm packages along with the browsers and drivers—to its GitHub Release. Before falling back to a lazy `npm install` or browser download, Doc Detective tries to restore the tarball that matches your exact platform and version, verifying it against a checksummed manifest.

The restore is strict and fails safe. It proceeds only when the release version, the platform (OS, architecture, libc, and OS major version), and every pinned dependency version match exactly, and when the downloaded archive's SHA-256 checksum matches its manifest. Any mismatch, checksum failure, or missing asset—for example, a platform with no published tarball—falls back silently to the normal lazy install, so you never end up with a broken cache, only a slower first run. Doc Detective also records failed restores so it doesn't re-download a known-bad asset on the next run.

The restore uses the same cache directory as the lazy install, so `DOC_DETECTIVE_CACHE_DIR` controls where restored assets land. In CI, the prebuilt cache speeds up cold runners and cache misses on its own. Persisting `DOC_DETECTIVE_CACHE_DIR` between runs (see [Runtime caching](/docs/ci/github-action#runtime-caching)) still avoids re-downloading across runs, and the two work together.

Three environment variables control this behavior:

| Variable | Effect |
|----------|--------|
| `DOC_DETECTIVE_PREBUILT` | Set to `0` (also accepts `false`, `no`, or `off`) to opt out of the prebuilt-cache restore entirely. Behavior is then identical to the lazy install. This is separate from `DOC_DETECTIVE_AUTOINSTALL`: `AUTOINSTALL` controls the postinstall pre-install, while `PREBUILT` controls only the tarball restore. |
| `DOC_DETECTIVE_PREBUILT_BASE_URL` | Override the base URL Doc Detective fetches tarballs and manifests from (a trailing slash is optional). Useful for air-gapped mirrors or internal artifact stores. Defaults to the GitHub Release download path for your installed version. |
| `DOC_DETECTIVE_PIN_BROWSERS` | Set to `1` (any value other than `0`, `false`, `no`, or `off`) to treat an already-installed browser and driver record as authoritative and skip the daily channel freshness re-check. Combined with a warm cache, repeat runs do no browser-related network requests. `--force` still bypasses the pin and reinstalls. |

<Note>

Downloads use the standard `HTTP_PROXY`, `HTTPS_PROXY`, and `NO_PROXY` environment variables. If those aren't enough for the download to succeed in your environment, the restore fails and falls back to the lazy install—it never blocks the run.

</Note>

## Auto-update

By default, Doc Detective checks the npm registry on startup and updates itself when a newer version is available:

- **Global installs** (`npm i -g doc-detective`): Runs `npm install -g doc-detective@<latest>` and re-executes.
- **npx invocations**: Re-executes with `npx -y doc-detective@latest`.
- **Local project installs**: Prints an "update available" message instead of modifying your `package.json`.

To disable auto-update, use any of these methods:

- Pass `--no-auto-update` on the CLI
- Set `autoUpdate: false` in your `.doc-detective.json` config file
- Set the `DOC_DETECTIVE_SKIP_AUTO_UPDATE=1` environment variable

CI environments skip the update check automatically whenever the `CI` environment variable has a value.
